# Домашнее задание 4

## Условие

Реализовать потокобезопасный IntSet на основе связного списка.

Последовательно реализовать 4 стратегии синхронизации:

- Грубая: Один мьютекс на весь список.
- Тонкая: Мьютекс на каждый узел (захват при проходе + ручная блокировка next перед удалением).
- Оптимистичная: Поиск без блокировок, затем захват и валидация.
- Ленивая: Пометка узла как удаленного (флаг) физическое удаление позже.

Исследование: Сравнить производительность (операций/сек) при 20% записи / 80% чтения.

Отчет: Вывод, при каких условиях «ленивая» синхронизация выигрывает.

## Исследование

См. бенчмарк в `main.cpp`

```
Threads: 24
10% add, 10% remove, 80% contains

Naive     : ops=407559, seconds=3.01436, ops/sec=135205
Fine      : ops=362450, seconds=3.00847, ops/sec=120476
Optimistic: ops=2651563, seconds=3.01592, ops/sec=879187
Lazy      : ops=5493170, seconds=3.02638, ops/sec=1815095
```

## Отчет

`Lazy` почти в два раза быстрее `Optimistic` и напорядок быстрее остальных. Вероятно потому,
что в `contains` нет блокировок. В `Optimistic` мы блокируем две ноды, и делаем дорогой
`validate`.

`Naive`, который для каждой операции захватывает мьютекс выглядит невыгодно в ситуации
когда потоков становится много, а `Fine` делает захваты слишком часто.

Вывод - `Lazy` выигрывает когда важна пропускная способность, а не стоимость
одной записи.
